# RealCap\.app 项目落地执行计划文档

# 一、项目核心目标

以最低成本完成 RealCap\.app 国际化冷启动，依托 GitHub Pages 托管多语言落地页，通过 SEO 引流积累用户，待市场验证后，通过二级域名切换至自有业务服务器，逐步搭建完整的“可信截图\+验真”业务体系，打造全球独有的真实截屏工具品牌。

核心关键词：低成本冷启动、多语言SEO、GitHub Pages托管、二级域名业务迁移、国际化市场、可信截图验真

# 二、核心架构规划（固定不变，贯穿全阶段）

## 2\.1 域名架构

- 主域名：realcap\.app（品牌核心，用于官网、落地页、SEO引流）

- 业务二级域名：app\.realcap\.app（后期切换至自有业务服务器，承载用户登录、截图上传、验真、数据存储等核心业务）

- 辅助二级域名（可选，后期扩展）：docs\.realcap\.app（帮助中心）、download\.realcap\.app（客户端下载）

## 2\.2 托管架构

- 第一阶段（冷启动）：realcap\.app 落地页/官网 → 托管于 GitHub Pages（免费，静态页面，支持多语言、HTTPS，适配SEO）

- 第二阶段（业务上线）：app\.realcap\.app → 指向自有业务服务器（承载动态业务，与主域名独立，不影响主域名SEO权重）

- 第三阶段（扩展）：逐步将辅助二级域名落地，统一品牌架构，提升用户体验

## 2\.3 业务边界划分

- GitHub Pages 仅承载：静态落地页、多语言切换、品牌介绍、功能说明、客户端下载引导、SEO内容（博客、帮助文档），不涉及任何动态业务（无登录、无上传、无数据存储）。

- 自有服务器承载：用户系统、截图上传、云端存证、验真功能、AI检测、客户端数据同步等核心动态业务，通过 app\.realcap\.app 对外提供服务。

# 三、分阶段执行计划（按优先级推进，可落地）

## 第一阶段：准备阶段（1\-2周，核心：域名\+托管\+基础页面）

### 3\.1 核心任务

1. 域名确认与DNS配置：

2. 确认 realcap\.app 域名已注册，完成域名实名认证（若未完成）。

3. 在域名管理后台，配置 DNS 解析：将 realcap\.app 指向 GitHub Pages（配置 A/AAAA 记录或 CNAME 记录，确保 HTTPS 生效，适配 \.app 域名强制 HTTPS 要求）。

4. GitHub 准备：

5. 创建 GitHub 仓库（命名建议：realcap\-app\-website），开启 GitHub Pages 功能，绑定 realcap\.app 主域名，确认 HTTPS 正常访问。

6. 配置仓库权限，仅核心成员可操作部署，避免误操作。

7. 添加禁止商用开源协议：选择合适的禁止商用开源协议（推荐 CC\-BY\-NC 3\.0 协议，该协议允许非商业用途的使用、复制、修改和分发，禁止用于商业盈利活动，且需保留原作者版权声明），在仓库根目录上传 LICENSE 文件，明确协议条款，标注 RealCap 相关版权信息，禁止他人将仓库内内容（含落地页代码、静态资源等）用于商业用途。

8. 基础页面开发：

9. 开发静态落地页核心模块：品牌介绍（RealCap 核心价值：可信截图、防AI伪造、可验真）、功能亮点、多语言切换入口、客户端下载引导（预留后期客户端下载链接）、联系我们、开源协议说明（简要标注仓库采用禁止商用开源协议，指引用户查看仓库 LICENSE 文件了解详情）。

10. 确保页面适配移动端、PC端，加载速度优化（压缩图片、简化代码），符合 SEO 基础要求（合理布局关键词、添加 meta 标签）。

### 3\.2 交付物

- realcap\.app 域名 DNS 配置完成，可通过 HTTPS 正常访问 GitHub Pages。

- GitHub 仓库搭建完成，落地页基础版本上传部署，可正常访问；仓库已添加禁止商用开源协议（LICENSE 文件），协议条款清晰。

- 落地页已添加开源协议简要说明，指引用户查看完整协议。

### 3\.3 风险与应对

- 风险1：\.app 域名 DNS 配置错误，导致 GitHub Pages 无法访问。应对：参考 GitHub Pages 官方文档，严格配置 A/AAAA 记录，配置完成后通过在线工具检测解析是否生效，若失败联系域名服务商协助。

- 风险2：HTTPS 无法生效。应对：GitHub Pages 自动生成 SSL 证书，耐心等待1\-24小时，若仍未生效，检查域名绑定是否正确，重新开启 GitHub Pages  HTTPS 功能。

- 风险3：开源协议选择不当或条款不清晰，导致商用侵权纠纷。应对：优先选用成熟的禁止商用开源协议（如 CC\-BY\-NC 3\.0），参考协议官方原文撰写 LICENSE 文件，明确标注禁止商用范围、版权归属，避免条款模糊；若有疑问，可参考开源协议官方文档或咨询相关专业人士。

## 第二阶段：冷启动阶段（3\-8周，核心：多语言\+SEO\+引流）

### 4\.1 核心任务

1. 多语言页面开发与部署：

2. 优先开发核心语种（覆盖主要目标市场）：英语（en）、西班牙语（es）、法语（fr）、日语（ja）、阿拉伯语（ar）、俄语（ru）。

3. 采用目录结构实现多语言：realcap\.app/en（英语）、realcap\.app/es（西班牙语），以此类推，统一页面布局，仅替换语言内容。

4. 添加多语言切换按钮，确保用户可自由切换，页面跳转流畅，无错乱。

5. 将多语言页面上传至 GitHub 仓库，完成部署，测试各语种页面访问稳定性；同步更新各语种页面的开源协议说明，确保与主页面协议表述一致。

6. SEO 优化（核心，决定冷启动引流效果）：

7. 关键词布局：针对各语种，布局核心关键词（如英语：trusted screenshot tool、anti\-AI fake screenshot、verifiable capture；其他语种对应翻译），合理分布在页面标题、meta 描述、正文、按钮文本中。

8. 内容优化：添加 SEO 友好内容，如博客板块（发布“如何识别AI伪造截图”“可信截图的应用场景”等干货文章）、帮助文档（多语言版本），提升页面权重；在帮助文档中新增“开源协议说明”章节，详细解读禁止商用条款，避免用户误解。

9. 外部引流：在海外社交平台（Twitter、LinkedIn、Reddit）发布品牌内容，链接指向 realcap\.app，积累外部链接，提升 SEO 排名；提交网站至 Google、Bing 等搜索引擎，加快收录。

10. 数据监测：添加 Google Analytics 代码，监测各语种页面访问量、跳转路径、用户停留时间，分析引流效果，优化关键词和页面内容。

11. 客户端准备（同步推进，不影响冷启动）：

12. 完成 Windows、macOS 客户端核心功能开发（窗口进程检测、微信/QQ 等应用可信验证、截图生成），浏览器插件开发（网页截图功能）。

13. 预留客户端下载链接，待客户端测试完成后，更新至落地页，引导用户下载。

### 4\.2 交付物

- 多语言页面全部部署完成，可正常切换、访问，适配各语种用户需求；各语种页面均包含开源协议说明。

- SEO 优化完成，核心关键词在 Google 等搜索引擎有基础排名，页面被正常收录；帮助文档已添加开源协议详细解读。

- Windows、macOS 客户端及浏览器插件核心功能开发完成，可进行内部测试。

- Google Analytics 数据监测正常，可查看用户访问数据。

### 4\.3 风险与应对

- 风险1：多语言翻译不准确，影响国际化用户体验和 SEO。应对：采用专业翻译工具\+人工校对，重点核对核心功能、关键词翻译，避免歧义；后期根据用户反馈优化翻译内容；同步校对各语种开源协议说明的翻译，确保条款表述准确，无歧义。

- 风险2：SEO 引流效果不佳，访问量过低。应对：优化关键词布局，增加高质量博客内容，加大海外社交平台引流力度，分析数据调整优化方向，耐心积累（SEO 见效周期通常为1\-3个月）。

- 风险3：GitHub Pages 流量达到软限制（100GB/月）。应对：冷启动阶段用户量少，流量远低于限制；若后期接近限制，可压缩页面资源、优化图片，或暂时关闭非核心语种页面，优先保障核心市场。

- 风险4：GitHub 判定页面违规，导致封号。应对：严格遵守 GitHub 服务条款，页面仅展示品牌、功能介绍，不涉及钓鱼、恶意取证、隐私侵犯等内容；明确 RealCap 正向定位（可信、防伪造、合规），避免违规风险；确保开源协议符合 GitHub 平台规范，不涉及违规条款。

## 第三阶段：业务上线阶段（8\-12周，核心：二级域名切换\+业务落地）

### 5\.1 核心任务

1. 自有业务服务器准备：

2. 搭建业务服务器（云服务器，优先选择海外节点，适配国际化用户访问速度），配置 HTTPS，确保服务器稳定、安全。

3. 部署核心业务系统：用户注册/登录、截图上传、云端存证（哈希计算、区块链存证）、验真页面、AI 检测接口、客户端数据同步接口。

4. 完成业务系统内部测试，修复 bug，确保核心功能（上传、验真、AI 检测）正常运行。

5. 二级域名配置与业务切换：

6. 在域名管理后台，配置 app\.realcap\.app 二级域名，指向自有业务服务器，完成 DNS 解析，测试 HTTPS 生效情况。

7. 更新 GitHub Pages 落地页内容：将“截图上传”“验真”等动态功能入口，全部指向 app\.realcap\.app，引导用户跳转至业务二级域名使用核心功能；同步更新落地页及各语种页面的开源协议说明，明确 GitHub 仓库内静态内容与自有业务服务器动态业务的协议区分（仓库内容禁止商用，业务服务按自有商业规则执行）。

8. 完成客户端与业务服务器对接，确保客户端截图后可正常上传至服务器，生成验真链接，实现全流程打通。

9. 用户引流与转化：

10. 通过 GitHub Pages 落地页（SEO 引流）、海外社交平台，引导用户下载客户端、访问 app\.realcap\.app 使用核心功能。

11. 优化验真页面体验，确保用户可快速验证截图真实性，提升用户留存。

12. 收集用户反馈，优化客户端功能和业务系统体验，解决用户使用过程中的问题；同步收集用户关于开源协议的疑问，及时优化协议说明内容。

### 5\.2 交付物

- 自有业务服务器搭建完成，核心业务系统部署上线，可正常运行。

- app\.realcap\.app 二级域名配置完成，与主域名联动，动态业务入口跳转正常；落地页及各语种页面的开源协议说明已更新，明确协议适用范围。

- 客户端与业务服务器对接完成，全流程（截图→上传→验真）打通，可正常使用。

- 实现用户引流与转化，积累首批种子用户，收集有效用户反馈（含开源协议相关疑问）。

### 5\.3 风险与应对

- 风险1：二级域名切换后，用户跳转异常、业务系统不稳定。应对：切换前进行充分测试，模拟用户跳转流程；切换后24小时内密切监测服务器状态、跳转情况，及时修复异常；提前告知用户切换事宜，减少用户困惑。

- 风险2：业务服务器负载过高、访问速度慢（国际化用户）。应对：选择优质海外云服务器，配置 CDN 加速；优化服务器性能，压缩数据传输，确保国际化用户访问流畅。

- 风险3：用户转化效果不佳，下载客户端后活跃度低。应对：优化客户端体验，简化操作流程；加强功能宣传，突出“可信验真”核心价值；推出用户激励机制（如免费验真次数），提升用户活跃度。

- 风险4：用户混淆开源协议适用范围，将业务服务误判为禁止商用。应对：在落地页、帮助文档中明确区分 GitHub 仓库静态内容（禁止商用）与自有业务服务器动态服务（商业服务）的边界，避免用户误解；针对用户疑问及时响应解答。

# 四、核心风险汇总与整体应对原则

## 4\.1 核心风险（重点关注）

1. GitHub 违规风险：核心规避点——不涉及钓鱼、恶意取证、隐私侵犯，明确 RealCap 正向定位，页面内容仅用于品牌和功能介绍；确保开源协议符合 GitHub 平台规范，不涉及违规条款。

2. SEO 引流不及预期：核心应对——长期坚持优化，多语种布局，优质内容输出，海外社交平台联动，耐心积累。

3. 二级域名切换异常：核心应对——切换前充分测试，切换后密切监测，做好用户引导和异常修复。

4. 国际化用户体验不佳：核心应对——精准多语言翻译，服务器海外节点部署，页面和客户端适配不同语种、不同设备；同步确保各语种开源协议说明翻译准确，无歧义。

5. 开源协议相关风险：核心应对——选用成熟的禁止商用开源协议，条款表述清晰，明确适用范围；及时解答用户疑问，避免协议误解和商用侵权纠纷；协议内容同步更新至各相关页面，保持一致性。

## 4\.2 整体应对原则

- 低成本优先：冷启动阶段依托 GitHub Pages 白嫖托管，不投入过多成本在服务器和推广上，聚焦核心功能和 SEO。

- 风险前置：每个阶段执行前，提前排查可能存在的风险，制定应对方案，避免被动；重点排查开源协议选择、条款表述、多语言翻译等相关风险。

- 数据驱动：全程通过 Google Analytics、用户反馈等数据，优化页面、SEO 和业务功能，不盲目推进；同步收集开源协议相关用户反馈，优化协议说明。

- 灵活调整：根据市场反馈和用户需求，灵活调整执行计划，优先落地核心功能，再逐步扩展；若开源协议相关条款需优化，及时更新 LICENSE 文件及相关页面说明。
